# 生成AI活用機械設計プラットフォーム ゴール定義書

## 1. 目的と適用範囲

本書は、生成AI活用機械設計プラットフォームの目標を**SMART基準**で定義し、成功/失敗の判定条件を明確にする。  
対象はPoCからProduction v1/v2までの継続的改善活動であり、単一プロジェクトの完了ではない。

## 2. SMART目標

### 2.1 Specific（具体的）

設計部門または機関内の設計レビュー案件を対象に、自然言語の設計依頼から、**監査可能なパラメトリック仕様とSTEP AP242/B-Rep、および必要に応じたSTL/OBJ/glTFの派生出力**を生成する基盤を構築する。

### 2.2 Measurable（測定可能）

- 構造化要件JSONの妥当率：99%以上
- 単一部品の初回CADコンパイル成功率：80%以上
- 仕様→STEP生成成功率：95%以上
- 幾何妥当性チェック通過率：98%以上
- 重要案件の人手レビュー完全実施率：100%
- 監査ログ欠損率：0%

### 2.3 Achievable（達成可能）

- PoC時点では単一部品・一工程に絞る
- 活用する技術は既存OSSまたは商用APIで取得可能
- 初期スコープではプリンタ実機依存の最終評価を義務付けない

### 2.4 Relevant（関連性）

この目標は、以下に直接つながる。

- 設計業務の記述品質と抜け漏れの削減
- 試作モデルのリードタイム短縮
- 設計意図・仕様・形状・検証結果のトレーサビリティ向上
- 法規・機密・知財リスクの統制強化

### 2.5 Time-bound（期限）

| Phase | 期限目安 | 完了条件 |
|---|---|---|
| PoC | 6–10週 | 単一部品のend-to-end成功 |
| Pilot | 追加10–16週 | 設計者が試行可能 |
| Production v1 | +4–6か月 | 部門導入可能 |
| Production v2 | +6か月以上 | 実案件適用可能 |

## 3. 成功基準

### 3.1 工学成功基準

| ID | 基準 | 判定方法 | 要件水準 |
|---|---|---|---|
| E1 | Requirement/Spec/DSL/Validationの4スキーマが固定されている | Schema contract test | Pass/Fail |
| E2 | 同一SPEC/DSLから同一artifact hashが再現可能 | Regression test | Pass |
| E3 | 単一部品で要件→STEP/STLのend-to-endが成功する | PoC run | Pass |
| E4 | 生成結果が幾何妥当性基準を満たす | Geometry check | Pass rate ≥ 98% |
| E5 | DFM/AMプロファイルを適用し、ルール違反を検出・修正可能 | DFM test | Pass/Fail |
| E6 | 生成物にtraceability IDが付与され追跡可能 | Artifact inspection | Pass |
| E7 | 法規/機密/輸出管理タグが案件に適用できる | Compliance audit | Pass |

### 3.2 運用成功基準

| ID | 基準 | 測定方法 | 要件水準 |
|---|---|---|---|
| O1 | 監査ログの欠損がない | Log audit | 0件 |
| O2 | 人手レビューが必要な工程が明文化されている | Review record | 100%実施 |
| O3 | 商業APIとオンプレ切替がポリシーで切替可能 | Deploy test | Pass |
| O4 | OWASP代表リスク（Prompt Injection、Sensitive Info Disclosure、Improper Output Handling）に対応 | Security test | Pass |
| O5 | 重要案件はexport-controlledルートで運用可能 | Compliance operation | Pass |

### 3.3 業務成功基準

| ID | 基準 | 測定方法 | 要件水準 |
|---|---|---|---|
| B1 | 単一部品の設計時間がPoCで一定量削減されている | Time measurement | Baseline比20%削減（暫定） |
| B2 | 試作前の仕様書/モデルの手戻りが減少 | Defect count | Baseline比減少（暫定） |
| B3 | 支配的コスト要素が特定されコスト計画が更新可能 | Cost accounting | Pass |
| B4 | 少なくとも1部門が業務内で使用を継続できる | Usage log | Pass |

## 4. 失敗条件（停止/見直し基準）

以下を満たす場合、計画を停止または設計を見直す。

- 要件妥当率がPoC開始後4週以内に80%を超えられない
- 単一部品でend-to-end成功が2連続で失敗する
- セキュリティ監査で主要リスク（外部流出/実行制御/権限制御）が通行できない
- 人手レビュー必須工程が特定できないまま本番導入しようとする
- 法規/安全保障/著作権/NDAの未解決リスクが放置される

## 5. 優先目標

| 優先度 | 目標 | 理由 |
|---|---|---|
| 1 | スキーマと実行分割の早期固定 | 以降の技術選定を安定させるため |
| 2 | PoCの高速完了 | 投資対効果を早期に検証するため |
| 3 | セキュリティ/法規フローの早期適用 | 後戻りのコストが大きいため |
| 4 | 人力依存工程の文書化 | 自動化と人間の関与を明確にするため |
| 5 | LLM活用範囲の順次拡大 | 運用リスクを小さく保ちながら適用拡大するため |
